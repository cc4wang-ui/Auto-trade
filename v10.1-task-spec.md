## 一、任務目標（一句話）

把 **私人信貸壓力訊號** 整合進 v10 的 Regime Filter，讓 HY spread 的急性升溫可以**強制升級** regime 到 CRISIS，避免在信用危機形成期進場做多。

## 二、為什麼要做（給 Claude Code 的 context）

### 觸發背景

- 2025/9 Tricolor + First Brands 雙重抵押詐騙
- 2025/12 BlackRock HPS 假應收帳款事件
- 2026/Q1 Apollo / Ares / Blue Owl 全部限贖
- BoE Bailey 公開警告「2008 前夕 déjà vu」
- 私人信貸市場 $2.1T，預估 2030 年翻倍

### 對 v10 的意義

v9/v10 原本的 Regime Filter 只看：

- VIX 期限結構（系統風險）
- 2Y ROC（政策意外）
- 同步性指數（跨市場壓力）

**遺漏**：信用市場急性壓力（HY spread 跳升）。
2008 年 Bear Stearns 對沖基金倒閉前 14 個月，HY spread 已經在升溫，但同步性還沒到 70。
v10 缺這個訊號 = 在信用危機形成期可能還在做多 = 災難。

### 補強目標

1. 新增「HY 急性升溫」偵測（一週 > 100bp）
1. 強制升級 regime（即使同步性 < 70）
1. Dashboard 顯示信用壓力等級
1. 新增 alert condition

## 四、輸出規格

### 變數定義（放在 Module 2 信用區塊之後）

```pine
// ═══ 私人信貸壓力子模組 ═══

// 1. 等級判定
string hy_pressure_level = na(hy_spread_val) ? "N/A" :
                            hy_spread_val > hyCrisisLevel ? "CRISIS" :
                            hy_spread_val > hyWarnLevel  ? "WARNING" :
                            hy_spread_val > 3.0          ? "ELEVATED" :
                            "NORMAL"

// 2. 一週急性跳升偵測（用 daily 週期的 [5]）
// ⚠️ Pitfalls P3-5: 不要把 array.get / 條件表達式寫成 ternary
float hy_5d_ago = nz(hy_spread_val[5], hy_spread_val)
float hy_weekly_jump = hy_spread_val - hy_5d_ago

bool hy_acute_event = hy_weekly_jump > hyAcuteJump and not na(hy_spread_val)

// 3. 對 regime 的影響旗標
bool hy_force_crisis = hy_pressure_level == "CRISIS" or hy_acute_event
bool hy_force_warning = hy_pressure_level == "WARNING" and not hy_force_crisis
```

### Regime Filter 整合（修改現有邏輯）

**找到現有的 regime 判定區塊**（v9 應該長這樣）：

```pine
string regime = vix_deep_backwardation ? "HALT" :
                us2y_roc_extreme       ? "SHOCK" :
                sync_index > sync_danger or vix_backwardation ? "CRISIS" :
                sync_index > sync_warn or stability_low       ? "WARNING" :
                "NORMAL"
```

**改成**（加入 HY 強制升級邏輯）：

```pine
// 原始 regime（基於同步性 + VIX）
string regime_base = vix_deep_backwardation ? "HALT" :
                     us2y_roc_extreme       ? "SHOCK" :
                     sync_index > sync_danger or vix_backwardation ? "CRISIS" :
                     sync_index > sync_warn or stability_low       ? "WARNING" :
                     "NORMAL"

// 信用壓力強制升級邏輯
string regime = regime_base
if hy_force_crisis and regime != "HALT" and regime != "SHOCK"
    regime := "CRISIS"
else if hy_force_warning and regime == "NORMAL"
    regime := "WARNING"

// 升級原因（給 dashboard 顯示用）
string regime_upgrade_reason = regime != regime_base ?
    (hy_acute_event ? "HY 急性跳升 +" + str.tostring(hy_weekly_jump, "#.##") + "%" :
     hy_pressure_level == "CRISIS" ? "HY 信用危機 (" + str.tostring(hy_spread_val, "#.##") + "%)" :
     "HY 信用壓力 (" + str.tostring(hy_spread_val, "#.##") + "%)") :
    ""
```

// regime 升級提示（如果有）

## 五、🔴 Pine Script 陷阱清單（必讀，違反就要重寫）

### 從 pattern-detector-pitfalls.md

- ✅ **P3-5**：`array.get()` 不能放在 ternary 條件裡 — 本次無 array 操作，但變數宣告順序要對
- ✅ **P3-2**：變數宣告順序 — `hy_spread_val` 必須在使用前已宣告
- ✅ **P3-1**：`ta.*` 函數必須在 global scope — 本次無使用，但 Dashboard 內計算要遵守

### 從 dev-guide.md

- ✅ **Gotcha #5**：table 行數預留餘量 — Dashboard 新增 3-4 row，確認 `table.new(..., rows=N)` 的 N 夠大
- ✅ **Gotcha #6**：`ta.correlation` NaN 防護 — 本次用 `nz()` 包 `[5]` 索引
- ✅ **Gotcha #2**：strategy 不支援 alertcondition — 區分 indicator vs strategy 版本

### 本次新增的潛在陷阱

- ⚠️ **`hy_spread_val[5]` 的時間軸**：hy_spread 是 D 週期 request.security，60min 主圖上 `[5]` = 5 個 daily bar = 一週交易日（OK）
- ⚠️ **na 防護**：FRED 數據可能延遲 1-2 天，必須 `not na()` 包所有計算
- ⚠️ **regime 升級的優先順序**：HALT/SHOCK 不被 HY 覆蓋（這兩個更嚴重）
