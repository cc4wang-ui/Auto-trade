# 技術型態偵測器 — 開發設計文件

## 需求還原

Cross 要一個獨立的 TradingView indicator：
- 自動判讀圖表上的技術線型型態
- 顯示符合度（品質分）和操作建議（進場/停損/目標）
- 適用任何標的、任何時間框架（TXF1! 分鐘/小時線為主要場景）
- 訊號品質 > 訊號數量
- 品質分必須隨 K 線即時更新（reactive）
- Fork 現有邏輯改造，不從零寫

## IDEO 設計框架

### ① Empathy — Cross 看圖時在想什麼

**5 秒問題**：「現在有型態嗎？」
→ dashboard 最上面一眼就能看到：有或沒有

**30 秒問題**：「這個型態值不值得做？」
→ 品質分 + 有效勝率 + R:R 一行看完

**2 分鐘問題**：「怎麼做？」
→ 進場價、停損價、目標價，圖表上畫線

**不要問的問題**：「為什麼這個品質分是 73？」
→ 底部小字公式就好，不要喧賓奪主

### ② Define — 核心問題

> 如何在任意時間框架上，用演算法可靠地偵測型態，
> 並且用即時 K 線數據動態評估型態品質？

關鍵挑戰：
1. Pivot 偵測必須產出交替序列（高-低-高-低），不能亂序
2. 型態匹配必須搜索序列中的幾何形狀，不是只看「最近 N 個」
3. 品質評分必須每 tick 更新，不能只在 K 線收盤時算
4. 過濾器要能區分「值得做的型態」和「噪音」

### ③ Ideate — 架構決策

#### A. Zigzag 統一序列（解決核心缺陷）

**舊架構（失敗的）**：
```
phP[] = [H0, H1, H2, ...]    ← pivot high 獨立存
plP[] = [L0, L1, L2, ...]    ← pivot low 獨立存
```
問題：phP[0] 和 plP[0] 之間沒有時間順序保證。
分鐘線上可能連續出現 3 個 pivot low 才出一個 pivot high。
型態條件假設 `l1b < h0b < l0b`（低-高-低交替），現實不成立。

**新架構**：
```
zzP[] = [19800, 20100, 19750, 20050, 19820, ...]  ← 價格
zzB[] = [bar1,  bar2,  bar3,  bar4,  bar5,  ...]  ← K線位置
zzD[] = [1,    -1,     1,    -1,     1,     ...]  ← 方向 (1=低, -1=高)
```
Zigzag 保證嚴格交替：低→高→低→高。
兩個連續同向 pivot 出現時，保留更極端的那個（兩低取更低、兩高取更高）。

#### B. 型態匹配：在 zigzag 序列上搜索

不再假設「最近 3 個 pivot 就是型態」。
而是搜索 zigzag 序列中最近 N 個節點（N=10~20），找符合幾何條件的子序列。

範例：雙重底需要 5 個 zigzag 節點
```
zzD: [1, -1, 1, -1, 1]  → 低-高-低-高-低（不對，需要 低-高-低）
```
實際上雙重底需要的是 zigzag 中的 3-node 子序列：
```
L1（低）→ H0（高=頸線）→ L0（低，與L1相似）
```
然後檢查 close > H0（突破頸線）。

#### C. 雙狀態：形成中 vs 已突破

| 狀態 | 條件 | dashboard 顯示 |
|------|------|---------------|
| 形成中 | 幾何條件通過，但 close 未突破觸發位 | 灰色，「待突破 ↗19850」 |
| 已突破 | 幾何條件 + close 突破觸發位 | 彩色，完整進場/停損/目標 |

這解決「偵測=0」的問題 — 大多數時候型態正在形成，
只有突破那一刻才變成 confirmed。
只顯示「已突破」的話，大部分時間畫面是空的。

#### D. 動態品質評分

4 因子，權重保持不變：
| 因子 | 權重 | 滿分條件（校準後） |
|------|------|-------------------|
| 對稱性 | 30% | 兩個對應 pivot 差距 < 容差的 30% |
| 深度 | 25% | 型態高度 ≥ 1× ATR |
| 突破力 | 20% | 收盤超過觸發位 ≥ 0.3× ATR |
| 量能 | 25% | 突破量 ≥ 1.2× 均量 |

形成中的型態：突破力 = 0（因為還沒突破），品質分天花板 ~75。
已突破的型態：所有因子都計算，品質分可達 100。

有效勝率 = 歷史基準 × (品質/100)

### ④ Prototype — 程式架構

```
Layer 1: 全域計算
  ├── ATR, 均量, volOK（每 tick 計算，不放 if 裡）

Layer 2: Zigzag 引擎
  ├── ta.pivothigh / ta.pivotlow 偵測原始 pivot
  ├── 合併成交替序列 zzP/zzB/zzD
  └── 保留最近 20 個節點

Layer 3: 型態搜索
  ├── 對每種啟用的型態，搜索 zigzag 序列
  ├── 幾何條件通過 → 標記「形成中」或「已突破」
  ├── 算品質分，存入結果 array
  └── adjTgt() 自動調整目標價確保 R:R ≥ minRR

Layer 4: 後處理
  ├── 品質過濾（品質 < minQuality → 砍）
  ├── 跨度/高度過濾
  ├── R:R 過濾（< 1.0 → 砍）
  ├── 冷卻期過濾
  ├── 排序（Tier → 有效 WR）
  └── 截斷（maxSignals）

Layer 5: 顯示
  ├── Dashboard（每 tick 刷新）
  ├── Label + Line（var + delete/recreate，reactive）
  └── Alert（只在「已突破」狀態觸發）
```

### ⑤ Test — 驗證計劃

1. **Pivot 可視化**：圖表上標記 zigzag 節點，確認交替正確
2. **TXF1! 分鐘線**：主要使用場景，必須有偵測結果
3. **TXF1! 小時線**：次要場景
4. **SPX 日線**：通用場景
5. **台股個股日線**：通用場景
6. **品質分回測**：在已知型態上（如 2022 年 SPX 頭肩頂）檢查品質分是否合理

## 噪音過濾參數（可調）

| 參數 | 預設 | 說明 |
|------|------|------|
| 最小跨度 | 5 根 | 型態從第一個 pivot 到最後一個 pivot 的距離 |
| 最小高度 | 0.5× ATR | 型態從頂到底的高度 |
| 最低品質分 | 70 | 低於此分不顯示 |
| 最低 R:R | 1.5 | 目標價自動調整確保 R:R ≥ 此值 |
| R:R 硬門檻 | 1.0 | R:R < 1 的型態直接不顯示 |
| 冷卻期 | 20 根 | 同型態不重複觸發 |
| 最大顯示數 | 3 | 同時只顯示最強 N 個 |

### R:R 處理邏輯
1. `adjTgt()` 函數：如果 measured move 算出的目標價 R:R < minRR，自動拉遠到 minRR
2. 後處理過濾：調整後 R:R 仍 < 1.0 的型態砍掉（型態結構太差，停損太遠空間太小）
